# tiny-agentic

> A UI-free, headless agentic framework for TypeScript/Node, built step by step learning from Claude Code.

## Development Workflow

This project follows a phased agent-driven development process organized by **scopes**:

- **Project scope (standard)** — whole-project workflow with 5 phases: `research` → `brainstorm` → `engineering` → `plan` → `implement`
- **Project scope (design-heavy, opt-in)** — 6 phases for design-driven products: `research` → `brainstorm` → `ux` → `engineering` → `plan` → `implement`. The `ux` phase produces a full UX spec (design language, components, screens, flows, a11y contract, microcopy) that the architect must respect. Chosen at `/phased-dev:init-project` time.
- **Feature scope (standard)** — adding a feature to this project, 4 phases (product framing + engineering collapse into one `engineering` phase): `research` → `engineering` → `plan` → `implement`
- **Feature scope (design-heavy, opt-in)** — UX-led features (onboarding rewrites, editor modes, search redesigns), 5 phases: `research` → `ux` → `engineering` → `plan` → `implement`. The `ux-designer` extends the existing project design system; the `feature-architect` then produces an engineering spec referencing the UX spec. Chosen at `/phased-dev:start-feature` time.

Every pipeline opens with a `research` phase: the `researcher` agent gathers prior art, technical feasibility, and domain constraints as evidence for the phases that follow. It makes no product or engineering decisions. For work on well-trodden ground, a short "no external research needed" doc satisfies the phase gate. (For this project, the research phase is satisfied by the pre-existing `docs/project/research/` subsystem map of the Claude Code reference — see Project-Specific Context below.)

Which pipeline a given scope uses is recorded in that scope's JSON under the `type` field (`project`, `project-design-heavy`, `feature`, or `feature-design-heavy`).

| Phase | Purpose | Output |
|-------|---------|--------|
| `research` | Gather evidence: prior art, technical feasibility, domain constraints, open questions. No decisions. | `docs/project/research/` (project) or `docs/feature/<name>/research/` (feature) |
| `brainstorm` | Define product: features, target users, edge cases, lite UX shape (project only) | `docs/project/brainstorm/` |
| `ux` *(design-heavy scopes only)* | Define visual & interaction design: language, components, screens, flows, a11y, microcopy | `docs/project/ux/` (project) or `docs/feature/<name>/ux/` (feature) |
| `engineering` | Define engineering: architecture, tech stack, interfaces. Respects upstream UX spec if present. | `docs/project/engineering/` (project) or `docs/feature/<name>/engineering/` (feature) |
| `plan` | Break work into tasks, define execution order. Coverage extends to UX deliverables when present. | `docs/project/plan/implementation-plan.md` (project) or `docs/feature/<name>/plan/implementation-plan.md` (feature) |
| `implement` | Execute tasks, write tests, verify, commit | `docs/project/tasks/` (project) or `docs/feature/<name>/tasks/` (feature) |

Scope state lives at `docs/.phased-dev/`:
- `state.json` — which scope is active
- `scopes/<id>.json` — pipeline and current phase for each scope

`docs/project/STATUS.md` is a **human-readable mirror** of the project scope's state, regenerated by commands. Agents read the JSON, not the markdown.

## Project-Specific Context

- **Reference source:** `claude-code-source-code/` is a git submodule containing the decompiled Claude Code v2.1.88 source. It is the learning reference for this framework — consult it, do not copy it wholesale.
- **Architecture research:** `docs/project/research/` contains a subsystem-by-subsystem map of the reference (see `docs/project/research/00-overview.md`). Read it before engineering decisions.
- **Hard boundary:** this framework is **UI-free and headless**. The core engine surfaces work as a typed event stream (async generator) and imports zero UI code. Any TUI/CLI/web UI is a *separate package* built on top — never inside this framework.
- **Provider goal:** support both Anthropic and OpenAI behind a provider abstraction the reference lacks (the reference is Anthropic-API-shaped throughout).

## How to Work on This Project

1. Run `/phased-dev:phase-status` to see what the active scope is doing.
2. Read the docs for the current phase and all prior phases.
3. Follow the agent-driven workflow: user provides direction, agents execute.
4. Document everything — each phase produces documentation, and the `implement` phase requires task-level execution logs.
5. Record important decisions in the scope's decisions log (project: `docs/project/decisions.md`; feature: `docs/feature/<name>/decisions.md`) with rationale.
6. Log known issues in `docs/project/known-issues.md` when deferring bugs or workarounds.

## Methodology (MUST READ)

This project has documented methodologies that all agents must follow:

- **Scope model** (`docs/.phased-dev/scope-model.md`) — how scopes, pipelines, and phase state work. Read this before writing or modifying any phased-dev command/agent.
- **Planning** (`docs/methodology/planning-methodology.md`) — How tasks are structured, ordered, and documented. Read this before writing or modifying a plan.
- **Execution** (`docs/methodology/execution-methodology.md`) — How tasks are implemented via the dev loop (implement → test → code review → fix) with sub-agent roles. Read this before executing any task.

These are not guidelines — they are requirements. Do not skip them.

## Conventions

- All documentation is in Markdown.
- One doc per task in `docs/project/tasks/` (project) or `docs/feature/<name>/tasks/` (feature), including execution logs and completion markers.
- Each task writes a `completion.md` after its commit lands — required for the implement phase to be considered complete.
- Each feature scope has its own `docs/feature/<name>/STATUS.md` mirror.
- User approves before advancing to the next phase, via `/phased-dev:advance-phase`.
- User can rewind to a previous phase via `/phased-dev:rewind-phase <phase>`.

## Phased-Dev Plugin Commands

This project was scaffolded with the `phased-dev` plugin. The full command set:

| Command | Purpose |
|---------|---------|
| `/phased-dev:phase-status` | Show active scope's current phase and what's next |
| `/phased-dev:list-scopes` | Show all scopes in this project |
| `/phased-dev:switch-scope <id>` | Switch the active scope (e.g., `project` or `feature/auth`) |
| `/phased-dev:start-phase` | Dispatch the agent for the active scope's current phase |
| `/phased-dev:start-task <NN>` | Run the per-task dev loop (during `implement` phase) |
| `/phased-dev:advance-phase` | Gated transition to next phase (requires user approval) |
| `/phased-dev:rewind-phase <phase>` | Rewind the active scope to a previous phase (resets status, truncates history) |
| `/phased-dev:generate-ux-preview` | Generate static HTML preview from UX spec (design-heavy scopes, any time after spec exists) |
| `/phased-dev:start-feature <name>` | Create a new feature scope (asks: standard or design-heavy) and dispatch its first-phase agent |
